# GROWTH — Triggers, Blueprints, Dead Ends

The seed installs only the engine. Everything below is built **when its trigger fires — never
speculatively**. The trigger column is the pain signal actually observed in the source project;
building an organ before its trigger is cargo-culting, and was itself a failure mode the source
project learned to avoid.

### Growth triggers

| Trigger (pain signal) | Organ to build | Blueprint |
|---|---|---|
| A work item outlives its session — an open item needs to survive the close and be findable later | `tasks/` with `TASK-NNN_description.md` files + template; session↔task cross-reference table | A1 |
| Open tasks accumulate past easy scanning; you need at-a-glance actionable/done/deferred views | View folders (`open/`, `done/`, `deferred/`) as **hardlinks** into `tasks/`, regenerated by a helper script; gitignored, local-only | A2 |
| The same structural mistake recurs across ≥2 session closes despite the template (malformed headers, bad enums, broken cross-references) | A lint script with **two modes**: enforce (staged files only, hard errors, gates the commit) and audit (whole repo, warnings only, never fails on immutable history) | A3 |
| The same manual toil recurs within a session (fiddly multi-step commands driven by hand) | A helper script under `helpers/` — this is the pain axis of the retrospective doing its job |
| Durable env-specific facts pile up inside session logs where nobody rereads them | `kb/` article per topic; distill at retrospective time (kb/ exists from day one — this trigger is about *starting to route into it* aggressively) |
| A procedure is executed twice and will recur (checklist-shaped, order matters) | `sops/` — one numbered SOP per procedure, with a template |
| The agent's training knowledge is thin on a vendor/technology and web docs are unreliable or offline | `reference/` corpus (converted vendor docs) + a retrieval protocol note in `kb/` + optionally a grounded retrieval subagent. Distilled facts still route to `kb/`; `reference/` is the bulky source layer |
| Documented state drifts from live reality between sessions (Tier 3 projects) | A periodic reconciliation runbook (`runbook/maintenance.md`): verify inventory against live state on a cadence |

### Paid-for dead ends (do not re-buy these)

- **Symlink view folders break under Windows+WSL dual access** — Windows Explorer cannot follow
  WSL symlinks. Use **hardlinks** + a regenerator script instead, and gitignore the view
  folders. *(Dormant: this repo is WSL/Linux-only — no Windows access expected.)*
- **Editing a view/copy instead of the canonical file** loses content or commits stale content.
  Convention beats tooling here: *edit the canonical file, then regenerate views*. A convention
  fully closed this failure mode; no tool was needed.
- **Hard-failing lint checks against immutable history deadlock the close.** A back-reference
  symmetry check that demands edits to an already-committed (immutable) session can never pass.
  Checks against immutable artifacts must be **warn-only**; hard checks apply only to files
  staged in the current commit.
- **A retrospective allowed to "carry forward" decays into a journal.** The source project ran
  months with soft retrospectives before making it a hard gate; the soft version produced notes
  nobody routed. Install it as a gate from day one (this seed does).
- **Per-item subfolders before there is per-item bulk** (one folder per device/unit each holding
  one file) collapse into flat files eventually. Start flat; grow a folder only when an item
  actually accumulates multiple artifacts.
- **A checklist box that can only be ticked after the commit that freezes it is unsatisfiable.**
  Define such boxes as "ticked at staging time" (the seed's session template already does).
- **`set -e` + bash `((var++))` aborts scripts on the first increment** (post-increment returns
  the pre-value 0 → exit 1) — after a destructive step may already have run. Use
  `var=$((var+1))`; audit helper scripts for this before trusting them with `rm`.

### Meta-principles for growing well (Tier 2, distilled from the source project's history)

1. **Structure earns its keep through use.** Let real work expose the shape; simplify
   aggressively when a pattern proves to be overkill.
2. **Mechanize late.** Build enforcement (linters, gates) only after the same drift recurs
   despite convention — but when you do build it, make it a hard gate, not advice.
3. **Fix the gate when it fights itself.** An enforcement mechanism that blocks a legitimate
   action is a bug in the mechanism; correct it immediately (usually: hard → warn for the
   over-reaching check).
4. **Cheapest durable fix wins.** A convention that fully closes a failure mode beats a tool
   that does the same; a tool beats a convention that keeps being violated.
5. **Identity discipline.** Refer to things by verified stable identifiers (live hostnames,
   account names, serials), never by convenient labels or display names — at every layer. The
   source project rediscovered this at two different layers before ratifying it as a principle.
6. **Every principle is incident-born and rationale-carrying.** Rule, why, how-to-apply, known
   exceptions — so the "why" survives independent of who wrote it. Do not ratify hypothetical
   principles.
7. **Single source of truth, thin shims.** When a platform quirk demands a second file (e.g.
   CLAUDE.md auto-loading), make it a one-line import, never a copy.

## Organ blueprints

When a growth trigger fires, build the organ from its blueprint below — do not reconstruct it
from memory.

### A1 — Task tracking

`tasks/_template.md`:

```markdown
# TASK-NNN — <short description>

**Opened:** YYYY-MM-DD
**Closed:** —
**Status:** open | in-progress | deferred | done | cancelled
**Items:** <subject>

## Sessions

| Session | Role |
|---------|------|
| [sessions/YYYY-MM-DD_topic.md](../sessions/YYYY-MM-DD_topic.md) | initiated |

<!-- Add rows as the task progresses. Role: initiated | worked | closed -->

---

## Context

<!-- Why does this task exist? What triggered it? -->

## Acceptance Criteria

- [ ] ...

## Notes

<!-- Progress updates, findings, decisions -->
```

Conventions that come with the organ: tasks live flat in `tasks/`; sessions and tasks
cross-reference each other via Role tables (`initiated | worked | closed`, comma-combinable);
the session template's Closing Checklist gains task steps (open items → tasks; touched tasks
cross-referenced). Artifact dirs `tasks/TASK-NNN_description/` follow the same naming as
session artifact dirs.

### A2 — View folders

`open/`, `done/`, `deferred/` at repo root, **hardlinks** to `tasks/` files, regenerated by
`helpers/tasks-links.sh` (idempotent: wipe views, re-link from each task's Status field).
Views are gitignored, local-only. **Convention: edit canonical `tasks/` files only, then rerun
the linker** — never edit or rename a view entry. (See dead-ends: symlinks, view-editing,
`((var++))`.)

### A3 — Lint gate

A validator script (e.g. `helpers/session-lint.sh`) with two modes:

- `--staged` — **enforce**: lints only files staged for the current commit; every finding is a
  hard ERROR; non-zero exit blocks the close. The git-commit checklist box must be `[x]` at
  staging time.
- `--all` — **audit**: whole-repo scan, warnings only, exit 0; immutable history is listed as
  debt, never edited and never a failure.

Checks: header fields present; enum values exact (`Type`, `Status`, `Role`); required sections
present; cross-reference targets exist. Cross-reference **symmetry** (the counterpart links
back) is warn-only even in enforce mode — the counterpart may be immutable. Skip artifact
subdirectories (`sessions/*/*`, `tasks/*/*`) — they are not templated files.

### A4 — Reference corpus (only if the trigger truly fires)

`reference/<vendor>/<doc>/` holding converted-to-Markdown vendor docs (source binaries
gitignored, converted text tracked, per-doc INDEX). Add a concept→aliases index once grep
vocabulary mismatch is observed (a search term missing while its synonym is present). Keep
`kb/` as the distilled layer; `reference/` is bulk. A grounded retrieval subagent (answers
only from the corpus, never parametric memory) is worth adding once the corpus exceeds what
ad-hoc grep serves well.
